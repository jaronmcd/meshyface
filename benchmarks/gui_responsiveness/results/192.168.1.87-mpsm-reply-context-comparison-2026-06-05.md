## Interpretation

This change is primarily user-experience cleanup, not a hot-path rendering optimization. It makes unresolved reply previews less noisy while keeping the original packet id and diagnostic detail in the tooltip.

The five-iteration full run is the most useful post-change signal: total p95 stayed essentially in line with the prior full run and long tasks remained at zero. The single-iteration full run had one noisy `poll:settings` sample, which is why its p95 is higher; the repeated run did not reproduce that as a broad regression.

The chat-only run loaded 955 recent chat rows and still reported zero long tasks. `switch:chat` remained effectively free, while `poll:chat` is still dominated by benchmark wait/frame timing rather than measured JavaScript long tasks.
