## Interpretation

- The original high first `switch:chat` sample was a benchmark artifact. Warmup left Chat already active, then the benchmark measured a redundant switch. The final run marks this as `alreadyActive: true` and skips the extra frame wait, reducing it to `0.0 ms`.
- Chat render instrumentation shows the large Chat samples are not feed DOM construction: render runs report `feedMarkupChanged: false`, `visibleMessages: 320`, and `totalMs: 0` under headless timer granularity.
- Forced poll measurements no longer re-apply the active layout before polling. This better matches the real periodic poll path.
- Long-task reporting now filters to each measurement window. The prior repeated long-task counts were Chrome buffered entries being counted in every sample.
- The signal heatmap now skips layer activation when Leaflet reports a zero-size map canvas. This fixed the headless benchmark failure from `leaflet-heat` calling `getImageData(..., 0, ...)`.

## Remaining Signal

The remaining high poll totals are mostly wall-clock or virtual-time wait around async polling, not clear main-thread rendering work. The best next target is poll phase instrumentation: fetch, parse, state normalization, render dispatch, and post-poll view-specific work should be timed separately before making another optimization.
